# AI学习助手设计文档（草案）

## 1. 文档目标

本文档用于定义 `KBSystemWeb` 项目中“AI学习助手”模块的产品目标、系统架构、核心流程、数据设计与迭代计划，作为后续研发、联调、测试和验收的基线文档。

## 2. 项目背景

当前项目已经具备以下基础能力：

1. 基于 Spring WebFlux 的流式对话能力
2. 基于 Spring AI 的模型接入能力
3. Redis 短期记忆能力
4. PgVector 长期向量检索能力
5. Agent + Tool 的基础执行框架

但从现状看，系统仍存在以下问题：

1. 系统提示词与“学习助手”场景不一致
2. 知识库内容未围绕学习场景构建
3. RAG 初始化链路未完全启用
4. 会话记忆未沉淀为“学习档案”
5. 评测体系偏检索召回，缺少学习效果指标

因此需要将现有 AI 能力从“通用实验型问答”升级为“面向学习场景的 AI 学习助手”。

## 3. 产品定位

### 3.1 产品目标

AI学习助手的目标不是简单回答问题，而是帮助用户完成完整学习闭环：

1. 理解知识点
2. 解答具体问题
3. 生成练习题
4. 分析错误原因
5. 记录学习进度
6. 提供复习建议

### 3.2 目标用户

1. 中学生/大学生
2. 有明确学习目标的自学用户
3. 需要阶段性辅导、答疑和复习支持的用户

### 3.3 核心价值

1. 比搜索更直接：给出结构化解释
2. 比普通聊天更专业：基于课程资料和知识库回答
3. 比一次性问答更持续：能记住学习过程并形成个性化建议

## 4. 功能范围

### 4.1 核心功能

1. 学习答疑
2. 知识点讲解
3. 题目解析
4. 练习题生成
5. 错题分析
6. 学习记录与总结
7. 基于资料的检索增强回答
8. 多轮上下文跟踪

### 4.2 非核心但预留功能

1. 学习计划自动生成
2. 学科模板切换
3. OCR 识题
4. 联网检索公开资料
5. 教师或管理员上传课程资料

### 4.3 本期不做

1. 自动判卷系统
2. 复杂多模态批改
3. 强依赖联网的实时搜索问答
4. 学习社区功能

## 5. 总体架构

### 5.1 架构分层

系统分为五层：

1. 接入层
   - AIController
   - SSE / 流式输出接口
   - 会话管理接口
2. 编排层
   - AgentService
   - ReActAgent / 后续可扩展 Planner Agent
   - Tool 调度逻辑
3. 能力层
   - ChatModel
   - RAG 检索
   - 学习记忆
   - 工具能力
4. 数据层
   - Redis 短期记忆
   - PostgreSQL / PgVector 长期记忆与知识库
   - 业务数据库中的用户与会话数据
5. 评测与运维层
   - Recall 评测
   - 回答质量评测
   - 日志、监控、限流、熔断

### 5.2 推荐模块图

```text
前端/客户端
   ->
AIController
   ->
AgentService
   ->
学习助手编排层
   -> ChatModel
   -> RAG检索
   -> 学习记忆服务
   -> Tool服务
   ->
Redis / PgVector / PostgreSQL
```

## 6. 核心业务流程

### 6.1 学习问答流程

1. 用户发起问题
2. 系统识别会话与学习场景
3. 加载短期对话记忆
4. 加载长期学习摘要与相关知识点
5. 对用户问题进行知识库检索
6. 组装 Prompt
7. 调用模型流式生成回答
8. 返回前端
9. 持久化本轮问答与学习摘要
10. 更新用户学习状态

### 6.2 错题辅导流程

1. 用户上传题目或输入题目内容
2. 系统识别题型、知识点、难度
3. 提供答案或引导式讲解
4. 记录错误原因和薄弱知识点
5. 写入“错题本记忆”
6. 后续在类似问题中优先提醒

### 6.3 练习生成流程

1. 用户指定学科、知识点、难度
2. 系统从知识库检索对应资料
3. 模型生成练习题
4. 输出答案与解析
5. 记录生成历史与用户作答结果

## 7. 模块设计

### 7.1 会话层设计

目标：支持一个用户在一个学习主题下连续学习。

建议抽象：

1. `conversationId`
2. `userId`
3. `subject`
4. `sessionType`
   - `QA`
   - `EXERCISE`
   - `REVIEW`
5. `learningGoal`
6. `currentTopic`

当前接口可保留流式能力，但建议从测试接口重构为正式业务接口：

1. `POST /api/ai-learning/chat`
2. `GET /api/ai-learning/chat/stream`
3. `POST /api/ai-learning/chat/stop`

### 7.2 Prompt 设计

Prompt 需要从“通用助手”升级为“学习助手”。

建议分为四段：

1. 角色定义
   - 你是一名严谨、善于启发式教学的 AI 学习助手
2. 行为约束
   - 优先基于知识库回答
   - 不确定时明确说明
   - 不直接跳步给结论
3. 输出规范
   - 知识点说明
   - 解题步骤
   - 易错点
   - 总结
4. 场景变量
   - 学科
   - 年级
   - 用户历史薄弱点
   - 检索资料片段

建议按场景拆成多个模板：

1. 学习答疑 Prompt
2. 题目讲解 Prompt
3. 练习生成 Prompt
4. 错因分析 Prompt
5. 阶段复习 Prompt

### 7.3 RAG 设计

目标：让回答基于真实课程内容，而不是模型自由发挥。

知识源建议包括：

1. 课程讲义
2. 教材章节
3. 习题解析
4. 错题总结
5. 学科知识点清单

RAG 设计建议：

1. 文档按“学科/章节/知识点/题型”建元数据
2. Chunk 不只按长度切分，要保留标题层级
3. 检索时增加过滤条件
   - `subject`
   - `grade`
   - `chapter`
   - `knowledge_point`
4. 支持父子块召回
5. 输出引用来源

建议元数据字段：

1. `subject`
2. `grade`
3. `chapter`
4. `section`
5. `knowledgePoint`
6. `questionType`
7. `difficulty`
8. `source`
9. `docType`

### 7.4 学习记忆设计

当前记忆更像对话记忆，后续要升级为学习记忆。

分为三层：

1. 短期记忆
   - 最近若干轮对话
   - 存在 Redis
   - 用于连续上下文
2. 中期记忆
   - 当前主题学习摘要
   - 当前章节掌握状态
   - 当前错题记录
   - 存在 PostgreSQL / PgVector
3. 长期记忆
   - 用户长期薄弱知识点
   - 历史练习表现
   - 阶段学习总结
   - 供后续个性化推荐使用

建议新增学习记忆对象：

1. `LearningProfile`
2. `WeakKnowledgePoint`
3. `WrongQuestionRecord`
4. `LearningSummary`

### 7.5 Agent 设计

当前 Agent 适合做基础工具编排，但学习助手不宜过度依赖通用 ReAct。

建议分两类模式：

1. 轻量问答模式
   - 直接 RAG + 回答
   - 适合普通知识问答
   - 延迟低
2. 学习任务模式
   - 用 Agent 编排步骤
   - 例如“先识别知识点，再检索，再讲解，再生成练习”
   - 适合复杂任务

建议 Agent 后续支持决策：

1. 是否需要检索
2. 是否需要查历史错题
3. 是否需要出题
4. 是否需要进入复习模式
5. 是否需要结束本轮任务

### 7.6 Tool 设计

建议工具围绕学习场景收敛，而不是泛工具。

本期建议工具：

1. `RetrieveKnowledgeTool`
   - 检索知识库资料
2. `GetLearningProfileTool`
   - 获取用户学习档案
3. `RecordWrongQuestionTool`
   - 记录错题
4. `GenerateExerciseTool`
   - 生成练习题
5. `FinishTaskTool`
   - 结束任务

后续可选：

1. `OCRQuestionTool`
2. `WebSearchTool`
3. `ScheduleReviewTool`

### 8.1 业务实体

建议增加以下实体：

1. 学习会话 `learning_session`

3. 错题记录 `wrong_question_record`
4. 学习摘要 `learning_summary`

## 9. 接口设计

### 9.1 核心接口

1. `POST /api/ai-learning/chat`
   - 发起学习问答
2. `GET /api/ai-learning/chat/stream`
   - 获取流式响应
3. `POST /api/ai-learning/chat/stop`
   - 停止生成
4. `POST /api/ai-learning/exercise/generate`
   - 生成练习题
5. `POST /api/ai-learning/wrong-question/analyze`
   - 错题分析
6. `GET /api/ai-learning/profile`
   - 获取学习档案

### 9.2 请求上下文

每次请求建议带：

1. `userId`
2. `conversationId`
3. `subject`
4. `grade`
5. `mode`
6. `prompt`

## 10. 非功能设计

### 10.1 性能

1. 普通问答首 Token 时间控制在 2 到 5 秒
2. 检索 `topK` 控制在合理范围
3. 记忆摘要异步化
4. 向量写入批量化

### 10.2 稳定性

1. 工具调用失败需降级
2. 检索失败可退化为无 RAG 回答并提示
3. SSE 中断后支持有限重连
4. 使用限流、熔断、超时保护模型调用

### 10.3 安全性

1. 学习资料按用户权限隔离
2. 上传文档内容校验
3. Prompt 注入防护
4. 敏感信息脱敏

## 11. 评测设计

### 11.1 检索评测

1. Recall@K
2. MRR
3. 命中章节准确率
4. 知识点命中率

### 11.2 回答质量评测

1. 正确性
2. 可理解性
3. 结构完整性
4. 是否贴合用户年级
5. 是否引用了正确资料

### 11.3 学习效果评测

1. 用户追问率
2. 错题重复率下降
3. 练习完成率
4. 用户满意度
5. 薄弱知识点覆盖率

## 12. 分阶段实施计划

### 阶段一：学习助手最小可用版

1. 替换现有系统提示词
2. 接入正式学习场景知识库
3. 启用知识库初始化
4. 打通 RAG + 流式回答
5. 增加学科、章节元数据过滤

### 阶段二：学习记忆版

1. 建立学习档案
2. 记录错题与薄弱点
3. 自动生成阶段摘要
4. 个性化上下文注入

### 阶段三：学习闭环版

1. 练习题生成
2. 错因分析
3. 复习建议
4. 学习进度可视化

## 13. 风险与注意事项

1. 当前知识库数据不对齐，先换数据再调 Prompt
2. 当前 Agent 与 Tool 注册不一致，需先统一
3. 当前部分流程仍是实验态接口，不宜直接作为正式产品接口
4. 记忆设计不能只存对话文本，必须沉淀学习结构化信息
5. 评测不能只看召回率，要看学习效果

## 14. 结论

本设计的核心思想是：以“学习闭环”为中心重构 AI 模块，而不是继续堆通用问答能力。系统应从“能回答”升级为“能辅导、能记录、能复习、能个性化”。
