# 简单个性化学习功能设计

## 1. 设计目标

当前阶段不引入复杂用户画像，不依赖大模型做深度推理，只实现一版轻量、稳定、可落地的个性化学习功能。

目标只有三点：

1. 让回答风格更贴合用户
2. 让会话更连续
3. 记住少量稳定偏好和学习目标

本设计优先保证体验，不影响主问答链路的首 Token 时间。

## 2. 设计原则

1. 优先使用用户显式输入，而不是依赖模型猜测
2. 优先使用规则提取，而不是复杂画像推理
3. 个性化信息分为长期稳定信息和短期会话信息
4. 主问答链路只读个性化信息，不做重更新
5. 个性化更新异步执行，失败不影响问答

## 3. 功能范围

本期只做四类个性化信息：

1. 学习目标
2. 讲解偏好
3. 当前学习主题
4. 简单薄弱点

本期不做：

1. 复杂用户画像
2. 自动水平评估
3. 多维学习推荐系统
4. 强依赖 LLM 的个性化总结

## 4. 个性化信息设计

### 4.1 学习目标

示例：

1. Java 后端面试
2. 操作系统期末复习
3. 数据结构入门

来源：

1. 用户手动设置
2. 用户在对话中明确表达，例如“我想准备 Java 面试”

### 4.2 讲解偏好

建议枚举：

1. `STEP_BY_STEP`
2. `CODE_FIRST`
3. `CONCEPT_FIRST`
4. `SHORT_ANSWER`

来源：

1. 用户手动设置
2. 规则识别
   - “详细一点” -> `STEP_BY_STEP`
   - “先给代码” -> `CODE_FIRST`
   - “先讲原理” -> `CONCEPT_FIRST`
   - “简单说” -> `SHORT_ANSWER`

### 4.3 当前学习主题

示例：

1. Redis
2. TCP
3. 二叉树

来源：

1. 当前会话最近几轮问题
2. 最近几轮中的高频主题词

该信息只作为短期会话状态，不直接进入长期画像。

### 4.4 简单薄弱点

示例：

1. 递归
2. 索引失效
3. 事务隔离

来源：

1. 用户明确表达“这里没懂”“还是不会”
2. 同一主题连续两次出现困惑表达

本期只做简单记录，不做复杂掌握度评分。

## 5. 数据设计

### 5.1 长期表 `learning_profile`

建议字段：

1. `id`
2. `user_id`
3. `subject`
4. `learning_goal`
5. `preferred_style`
6. `preferred_language`
7. `weak_points_json`
8. `updated_at`

说明：

1. `learning_goal` 保存当前主目标
2. `preferred_style` 保存当前讲解偏好
3. `weak_points_json` 保存简单薄弱点列表

### 5.2 短期表 `learning_session_personalization`

建议字段：

1. `conversation_id`
2. `user_id`
3. `current_topic`
4. `recent_topics_json`
5. `updated_at`

说明：

1. 该表只反映当前会话的短期主题
2. 会话结束后可继续保留，也可按 TTL 清理

### 5.3 异步任务表 `learning_profile_task`

建议字段：

1. `id`
2. `conversation_id`
3. `request_id`
4. `turn_index`
5. `status`
6. `owner_instance`
7. `lease_expires_at`
8. `retry_count`
9. `available_at`
10. `last_error`
11. `created_at`
12. `updated_at`

说明：

1. 复用当前 `learning_session_memory_task` 的设计思路
2. 每轮对话结束后投递一条轻量个性化更新任务

## 6. 更新规则

### 6.1 学习目标更新

仅在以下情况更新：

1. 用户手动设置目标
2. 用户明确表达目标

示例规则：

1. 匹配“我想学”
2. 匹配“我要准备”
3. 匹配“我现在在学”

更新策略：

1. 直接更新 `learning_profile.learning_goal`
2. 记录更新时间

### 6.2 讲解偏好更新

仅在以下情况更新：

1. 用户手动设置偏好
2. 用户明确表达偏好
3. 同类表达重复出现达到阈值

示例规则：

1. “详细一点” -> `STEP_BY_STEP`
2. “给我代码” -> `CODE_FIRST`
3. “先讲原理” -> `CONCEPT_FIRST`
4. “简单一点” -> `SHORT_ANSWER`

阈值建议：

1. 显式表达一次即可更新
2. 隐式表达连续两到三次再更新

### 6.3 当前学习主题更新

每轮对话结束后，从当前用户问题中提取主题词。

更新策略：

1. 写入 `learning_session_personalization.current_topic`
2. 追加到 `recent_topics_json`
3. 最近几轮重复出现的主题优先作为当前主题

### 6.4 简单薄弱点更新

示例规则：

1. 匹配“没懂”
2. 匹配“还是不会”
3. 匹配“为什么这里这样”
4. 匹配“总是搞混”

更新策略：

1. 先定位当前主题
2. 同一主题连续两次出现困惑表达后，加入 `weak_points_json`
3. 同一薄弱点已存在则不重复写入

## 7. 与现有代码的结合方式

### 7.1 读取个性化信息

建议在主问答链路中新增 `ProfileContextService`，在构建 Prompt 前加载：

1. `learning_profile`
2. `learning_session_personalization`

然后将结果摘要注入 `AiLearningApplicationService.buildSystemPrompt(...)`。

当前可复用接入点：

1. `SessionStorageService.loadSnapshot(...)`
2. `AiLearningApplicationService.buildSystemPrompt(...)`

### 7.2 更新个性化信息

建议在 `SessionStorageService.appendTurn(...)` 成功后，新增一条 `learning_profile_task`。

后台新增一个与 `SessionMemorySynchronizationService` 类似的异步服务，例如：

1. `LearningProfileSynchronizationService`

职责：

1. 读取本轮用户问题
2. 使用规则提取目标、偏好、主题、困惑信号
3. 更新短期主题
4. 更新长期档案

### 7.3 Prompt 注入建议

在系统提示词中新增一段简短摘要：

```text
用户个性化信息:
- 学习目标: Java 后端面试
- 讲解偏好: 先讲概念，再给代码示例
- 当前主题: Redis
- 已记录薄弱点: 递归, 事务隔离
```

要求：

1. 内容要短
2. 不要拼接过多历史
3. 缺失字段可直接省略

## 8. 接口建议

为降低模型依赖，建议增加用户可手动设置个性化信息的接口。

建议接口：

1. `GET /api/ai-learning/profile`
2. `POST /api/ai-learning/profile`

`POST` 支持设置：

1. `learningGoal`
2. `preferredStyle`
3. `preferredLanguage`

这样系统只负责做轻量补充，核心目标和偏好由用户控制。

## 9. 体验保证

1. 主链路只读取个性化信息，不做重更新
2. 规则提取放在回答完成后异步执行
3. 个性化更新失败不影响问答结果
4. 不依赖联网搜索
5. 不依赖复杂 LLM 推理

## 10. 实施顺序

建议按以下顺序落地：

1. 新增 `learning_profile`
2. 新增 `learning_session_personalization`
3. 新增读取个性化摘要的服务
4. 在 Prompt 中注入个性化摘要
5. 新增 `learning_profile_task`
6. 新增异步规则更新服务
7. 最后补 `GET/POST /api/ai-learning/profile`

## 11. 结论

当前项目更适合先做“简单个性化学习功能”，而不是复杂用户画像。

这版方案的核心是：

1. 用户手动设置目标和偏好
2. 系统规则提取当前主题和简单薄弱点
3. 主链路只读，异步更新
4. 在不增加明显复杂度的前提下提升学习体验

这样可以先把个性化功能做稳，再决定后续是否引入更复杂的画像能力。
